C) 修复 HI API 的越权入口（必须）
------------------------------------------------------------
在 api/api/hi_api.py 里做如下修改：

0) 新增通用 helper（必须）：
   - def hi_scoped_query(model): 对 model 应用 hi_filter_by_tenant + 额外规则（比如 HostingSite/Miner 需要自定义）
   - def hi_get_scoped_or_404(model, id_field_name, id_value): 只能从 scoped query 取一条

1) /miners：
   - tenant 用户：只允许看到 Miner.hi_tenant_id == g.hi_tenant_id
   - operator 用户：只允许看到 org 下所有 tenants 的 miners（通过 HiTenant.org_id == g.hi_org_id 获取 tenant_ids）
   - tenant 用户传 tenant_id 参数：忽略或若不等于 g.hi_tenant_id 则 403

2) /groups：
   - tenant 用户：强制 tenant_id = g.hi_tenant_id（忽略 query 里的 tenant_id）
   - operator 用户：仍可按 tenant_id 过滤，但只能过滤到本 org 内 tenant

3) /sites（当前 list_sites 会返回所有 hosting_sites，必须修）：
   - operator：只返回 hosting_sites.hi_org_id == g.hi_org_id
   - tenant：只返回“与本 tenant 有资源关联”的站点：
     方案：从 Miner 取 distinct site_id where hi_tenant_id == g.hi_tenant_id，再查 HostingSite.id in (...) 且 hosting_sites.hi_org_id == g.hi_org_id

4) /contracts、/usage、/invoices：
   - 所有 list 类接口：tenant 用户不得通过 query 参数 tenant_id 越权（若传了且 != g.hi_tenant_id => 403）
   - generate_usage：
     - tenant 用户：忽略 body.tenant_id，强制使用 g.hi_tenant_id
     - operator：允许传任意 tenant_id，但必须属于 g.hi_org_id

5) Curtailment 相关（simulate/execute/verify/report/command_status）：
   - 任何通过 plan_id / request_id 获取资源的接口，必须 scoped：
     - tenant：仅允许 plan.tenant_id == g.hi_tenant_id（不允许 tenant 访问 site_wide plan，避免看见其他租户）
     - operator：允许 org 内全部
   - /curtailment/command_status：按 request_id 查询时也必须加 tenant/org scope（tenant 必须 tenant_id == g.hi_tenant_id；operator 必须 org_id == g.hi_org_id）
